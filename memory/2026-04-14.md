## DeckBoss Hardware Strategy (02:16 UTC)

### The Product Stack
- **DeckBoss** — the physical box (Jetson-based) that plugs into a boat
- **Cocapn** — the software agent inside the DeckBoss
- **Plug and play** — immediate usefulness out of the box
- **Minimum viable:** second navigation + chatbot with perfect memory (time/location stamped)

### Growth Path
1. **Day 1:** Second nav + chatbot. Already useful.
2. **Week 1:** Captain connects more boat systems. Cocapn wires the software, instructs human on hardware/wiring puzzles.
3. **Month 1:** Template applications accumulate from successful integrations. Off-the-shelf options grow.
4. **Beyond:** When compute demand exceeds one Jetson, system recommends hardware/cloud expansion with pros/cons.

### Multi-Unit Fleet on One Boat
- Captain links multiple DeckBoss units (headless)
- Each runs its own Cocapn
- One unit is the **Commodore** — the chatbot the human talks to
- Others defer to Commodore status
- Scale by adding boxes, not upgrading boxes

### Oracle1's Role
- Bigger project than Casey + JC1 combined
- Building the cloud infrastructure, fleet protocols, standards, research
- The global system that makes every DeckBoss update smarter
- Casey and JC1 eventually prune to the actual hardware product
- Oracle1 keeps making the backend better

### JC1's Path
- Eventually pruned to DeckBoss hardware actualization
- His Jetson work becomes the prototype
- capitaine becomes the on-device agent
- starship-jetsonclaw1 becomes the real telemetry bridge
